# Trojan War Naming Convention

Descriptive names remain authoritative; mythic names are memorable aliases.
This prevents a contributor from needing classical knowledge to operate the
project.

| Alias | Use | Reason |
|---|---|---|
| **Laocoön** | adversarial review of a seemingly beneficial integration | warned against bringing the horse inside the boundary |
| **Cassandra** | research radar and early-warning signals | saw danger early but still required others to act |
| **Hector** | privacy, custody, and defensive-boundary gate | defender of Troy |
| **Odysseus** | counterfactual strategy and deceptive-path analysis | associated with the horse strategy; use only for red-team work |
| **Ajax Line** | collective professional-knowledge sovereignty and sector hold-the-line research | Ajax is a memorable defender associated with holding the Achaean line; the plain descriptive thesis remains authoritative |
| **Iris** | routing, notices, and contributor handoffs | messenger across actors and domains |
| **Priam** | named human governance/release decision | final authority remains human, not automated |

Rules:

1. File names, schema IDs, and API fields use plain descriptive language.
2. An alias never carries authority or epistemic status.
3. Do not name a real company, product, or person after an antagonist.
4. Reserve “Trojan horse” for a defined threat hypothesis, not a factual label.
5. New aliases require a one-sentence operational meaning and owner approval.

Recommended future public repository name: **Laocoon Ontology Observatory**.
It signals warning and investigation without asserting misconduct. Two shorter
alternatives are **Cassandra Ontology Radar** and **Hector Knowledge Boundary**.
Renaming remains a repository-owner decision.
